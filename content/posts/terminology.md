# Function Calling

## 1. 解决了什么问题？

在没有 Function Calling 之前，LLM 存在三个核心痛点：

1. 信息滞后（时效性差）： 模型的知识停留在训练数据截止日期。比如你问“现在的天气”或“昨天的股市”，它无法准确回答。
2. 幻觉问题（不擅长精确计算）： LLM 本质是概率预测，让它做复杂的数学计算、查询数据库或处理逻辑严密的任务时，很容易“一本正经地胡说八道”。
3. 无法与外部世界联动（静态系统）： 模型被困在对话框里。它不能帮你发邮件、订机票或操作你的数据库。

**核心转变：从“文本生成”到“意图识别”**
有了 Function Calling 后，模型不再尝试硬背答案，而是转变为一个智能调度中心：

- 以前： 用户问 -> 模型尝试用现有知识回答 -> 可能是错的。
- 现在： 用户问 -> 模型判断是否需要查工具 -> 输出函数名称和参数 -> 开发者执行工具 -> 模型根据返回结果总结答案。

## 2. Function Calling的工作流程
理解这个流程至关重要，因为它是一个多轮往返的过程。

1. 定义工具： 开发者告诉模型：“我有这些函数（如 get_weather），它们的作用和参数是什么。”
2. 用户提问： 用户输入指令（如“上海今天多冷？”）。
3. 模型决策： 模型发现需要外部信息，返回一个特殊的响应，包含：函数名（get_weather）和参数（{"location": "Shanghai"}）。
4. 本地执行： 你的程序解析这个响应，在自己的服务器上运行实际的代码。
5. 反馈结果： 你的程序将函数运行的结果（如 {"temp": "5°C"}）发回给模型。
6. 最终回答： 模型根据这个结果，组织语言回答用户。
